What problem does a mixer actually solve?

On a public blockchain, transfers are visible. If one address funds another address, the relationship can often be followed. A mixer breaks this direct link by replacing “this address paid that address” with a private note system.

The important part is not only hiding one transaction. The system also has to avoid leaking information while storing notes, fetching Merkle paths, syncing events, generating proofs, and submitting transactions.

Purpose

The project protects the whole path around a private payment.

A mixer contract alone is not enough. Privacy can fail in the client, in local storage, in Merkle path lookup, in recovery infrastructure, or in the way a proof is prepared. This site explains each of those layers in the order a user actually meets them.
Problem

Public chains create visible links.

If a user deposits from one Stellar account and later withdraws to another account, an observer should not be able to trivially connect those two actions.

Mechanism

Private notes replace direct balances.

The mixer does not work like a normal wallet balance. It creates notes. The public chain sees commitments and nullifiers; the user keeps the private note data locally.

Boundary

Servers help without owning the user.

TreePIR, the event server, and the Groth16 wrapper are helper services. Each one has a narrow job and should not become a custodian of funds or a holder of private note material.

Desktop application

The interface is where privacy becomes usable.

The desktop app is not just a visual shell. It is the place where Mixer Identity, notes, accounts, proof inputs, archive sync, and transaction submission come together.

Note model

A note is the private unit of value inside the mixer.

A note is not just “some database row”. It is the object that lets the user later prove ownership, spend privately, create a recipient note, or withdraw without revealing which earlier deposit was used.

Why flexible notes matter

Many mixers use fixed denominations. That can simplify some analysis, but it makes ordinary payments awkward. Stellar Privacy Mixer is designed around flexible-value notes so deposits, transfers, change, and withdrawals can feel closer to normal payments.

Why ownership stays private

The note owner does not publish the full note. Instead, the client constructs proof inputs locally. The proof says, in effect: “I know a valid note in the tree, and I am allowed to spend it,” without revealing which historical note is being spent.

Why nullifiers are public

The contract must reject double spends. A nullifier gives it a public marker for “this note has already been used” without forcing the contract to learn the plaintext note or link it directly to the deposit.

Walkthrough

The system becomes clear when you follow one note.

A deposit creates a note. A transfer creates another note. A withdrawal spends a note. Recovery helps the desktop rebuild local state from encrypted public history.
01

Deposit creates a private note.

The desktop creates private note data locally, publishes only a commitment to the Soroban mixer contract, and stores the full note inside the encrypted local vault.

  • The user chooses an amount.
  • The client creates private note material.
  • The contract receives only a public commitment.

The public chain learns that a commitment was added. It does not learn the private note contents that will later be used to spend.

Privacy boundaries

What each layer is allowed to learn.

This is the key idea. A clean privacy system is not magic; it is careful separation.
Layer What it sees What it should not learn
Desktop app Private notes, Mixer Identity, account metadata, proof inputs. It should not leak this material into ordinary remote services.
Soroban contract Commitments, roots, nullifiers, verification results, token movement. Plaintext notes, user vault data, and private ownership material.
TreePIR server Tree data and PIR protocol messages. The exact note leaf path the user is preparing to spend.
Event server Encrypted notes, nullifier history, sync state. Plaintext note contents or who can spend a note.
Groth16 wrapper Receipt wrapping input and output format. Stellar secret keys, wallet state, or vault passwords.

Merkle path retrieval

Why TreePIR matters.

To spend a note, the client needs Merkle path data. A naïve server request would reveal too much. TreePIR is used so path retrieval does not become a simple tracking signal.
Naïve lookup

“Give me the path for leaf 22.”

The server immediately learns which leaf the user is interested in. If that leaf is later spent, this can become useful correlation data.

instead
TreePIR-style retrieval

“Help me retrieve path data without seeing my selected leaf directly.”

The server still provides tree data, but the request is structured so the selected path is not handed over as a plain index.

Infrastructure

Each helper has one narrow job.

The goal is to make every part replaceable and understandable. A helper should help; it should not become the place where privacy assumptions silently accumulate.

Mixer contract

The contract is intentionally small. It stores commitments, root history, spent nullifiers, verifier configuration, and token movement rules. It does not store plaintext notes or local wallet state.

Future directions

The same structure can support more than withdrawals.

These are extensions the architecture is shaped to support. They are written as future directions, not as a claim that every flow is already production-ready.

Private service payments

Notes can be extended with structured payload ideas so a service can recognize what was paid for without learning the payer identity.

Infrastructure incentives

TreePIR servers, archive mirrors, proof wrappers, and future relayers can become paid parts of the privacy network.

Fee-payer relays

Stellar fee-payer patterns can reduce the first-funding link by letting relayers pay transaction fees for users.

Tor-assisted submission

The most sensitive network action is transaction submission. Future clients can route that step separately while keeping sync fast.

Open-source repositories

One system, split into inspectable components.

The repositories are separated so a reviewer can inspect the contract, proof code, TreePIR code, event archive, wrapper service, and desktop client independently.
