Overview

Privacy is only real when the whole path is designed carefully.

A mixer can lose privacy far away from the contract: when secrets are stored locally, when Merkle paths are fetched, when recovery happens, when proofs are prepared, or when a transaction first enters the network. Stellar Privacy Mixer treats those as first-class design problems.
01

On-chain surface stays small

The Soroban contract deliberately avoids storing a full tree or bulky user state. It keeps only the information that must be public: note commitments, root history, spent nullifiers, proof verification, and token movement rules.

02

Private data stays local

Notes, Mixer Identity material, recovery data, and account metadata are meant to stay on the user's device inside an encrypted local vault, instead of being treated as ordinary app state.

03

Infrastructure is split into narrow roles

TreePIR, the event server, and the Groth16 wrapper are helpers with sharply limited jobs. That makes the system easier to reason about and easier to scale without handing too much power to one service.

What is a note?

A note is the private unit of value inside the mixer.

The mixer does not operate like a traditional wallet balance. Value is represented as private notes. A note is constructed so it can be committed into a Merkle tree, later proven to exist, and then spent without publicly exposing which earlier deposit it came from.

Flexible by amount

Notes are not restricted to fixed denominations. A user can deposit, transfer, and withdraw arbitrary amounts, which makes the system more usable than mixers that force everyone into a small set of fixed note sizes.

Owned by secret material

Each note has an owner. Only the holder of the required private material can assemble valid proof inputs for spending it. In practice, that means only the legitimate owner can produce the STARK proof that is later wrapped into the Groth16-compatible form accepted by the verifier stack.

Useful beyond person-to-person transfers

Notes can also carry structured payload ideas for future service-payment flows. That opens the door to private payments for goods, APIs, or infrastructure, where the receiver learns what was paid for without learning who paid.

Privacy boundaries

Who sees what?

The design becomes easier to trust when each layer is allowed to learn only what it actually needs.
Local desktop

Sees the private note.

The desktop creates and stores the private note, account metadata, recovery data, and proof inputs. This is the sensitive side of the system, so local protection matters.

Mixer contract

Sees public commitments and spends.

The contract sees note commitments, root history, nullifiers, proof verification, and token movement. It does not see plaintext note contents or the user's local vault.

Helper servers

See only narrow operational slices.

TreePIR helps with private path retrieval, the event server mirrors encrypted note data and nullifiers, and the wrapper assists proof-format conversion. None of them should become custodians of user funds.

01

Deposit creates a private note.

The desktop creates private note data locally, publishes only a commitment to the Soroban mixer contract, and stores the full note in the encrypted local vault.

  • Creates note data locally
  • Publishes a public commitment on-chain
  • Keeps plaintext note contents on the device

Infrastructure

Each helper has one narrow job.

Narrow responsibilities make the system easier to inspect, easier to replace in parts, and easier to scale into a broader privacy ecosystem.

TreePIR Server

Merkle path lookup can leak which note a user is preparing to spend. TreePIR helps separate path retrieval from direct leaf disclosure so the server is not simply handed the exact selected leaf index.

Future directions

The current system is a foundation, not the endpoint.

The architecture is intentionally shaped so more privacy-preserving services can be layered on top of it later.

Private service payments

Payload-bound note flows can eventually let a service recognize what was paid for while still not learning who paid.

Infrastructure incentives

TreePIR servers, archive mirrors, wrappers, and relays can become economically incentivized parts of a broader network.

Fee-payer relays

Stellar fee-payer patterns can help users interact with the mixer without exposing a direct first-funding link.

Tor-assisted submission

Only the sensitive transaction-submission step may need private transport. Sync and reading can stay lightweight.
