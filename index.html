Designed around the weak points

Privacy is more than a contract.

The contract is only one layer. Real privacy also depends on how the client stores notes, retrieves Merkle paths, syncs events, builds proofs, and submits transactions.

Encrypted local vault

Mixer Identity, private notes, and account data stay protected locally with OS keyring-backed storage.

Private Merkle retrieval

TreePIR lets the client fetch Merkle path data without revealing the selected note path to the server.

Minimal on-chain surface

The Soroban contract focuses on commitments, roots, nullifiers, proof verification, and token movement.

Recovery and sync

Event infrastructure indexes encrypted notes and nullifiers so clients can sync and recover state.

How it works

Simple user flow. Strong privacy boundaries.

01

Deposit creates a private note.

The client creates private note data, publishes a commitment to the mixer contract, and stores the full note inside the local encrypted vault.

Infrastructure

Each server has one narrow job.

Supporting services improve availability and usability without becoming custodians of user funds or holders of private note material.

TreePIR Server

Merkle path lookup can leak which note a user is preparing to spend. TreePIR separates path retrieval from direct leaf disclosure, reducing server-visible correlation.

Future directions

Built to grow into a wider privacy ecosystem.

Service payments

Payload-bound notes can support private payments for invoices, API access, and digital services without exposing payer identity.

Infrastructure incentives

TreePIR servers, event mirrors, wrappers, and future relayers can be rewarded by the same payment layer.

Fee-payer relays

Relayers can help abstract Stellar fees and reduce first-funding links while receiving private compensation.

Tor submission

Transaction submission can be routed separately while ordinary public sync remains lightweight and fast.
